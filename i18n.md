# Internationalization (i18n) Plan for Silent

## Goal

Make Silent fully i18n-ready with a maintainable translation workflow, while preserving the current architecture (Qt/C++ UI + Rust backend via CXX FFI).

## Guiding Principles

- All user-facing text is localized in Qt/C++.
- Rust stays language-neutral (codes/enums/structured data where possible).
- Locale affects display only (formatting), not internal protocol/parsing logic.
- Roll out incrementally so the app stays buildable and testable after each phase.

## Phase 0: Baseline Audit

1. Inventory all user-visible strings in `src/`:
   - labels, placeholders, button text
   - modal titles/messages
   - validation warnings/errors
   - tab names, tooltips, status text
2. Categorize findings:
   - static UI strings
   - dynamic formatted strings
   - Rust error passthrough strings
   - non-user-facing/debug strings
3. Create a file-by-file migration checklist, prioritizing:
   - `src/MainWindow.cpp`
   - `src/screens/Send.cpp`
   - `src/screens/Receive.cpp`
   - `src/screens/Settings.cpp`
   - `src/screens/History.cpp`
   - `src/screens/modals/*.cpp`

## Phase 1: Translation Build Pipeline (Qt Linguist)

1. Add Qt translation targets in `CMakeLists.txt`:
   - use Qt6 translation workflow (`qt_add_translations` or equivalent)
   - ensure `lupdate` and `lrelease` are part of dev/release flow
2. Create translation sources under `i18n/`:
   - `i18n/silent_en.ts` (source-language catalog)
   - one pilot locale, e.g. `i18n/silent_es.ts`
3. Compile and package `.qm` files for runtime loading:
   - include in resources or deploy alongside the binary
4. Document commands for contributors:
   - update sources, open in Qt Linguist, release compiled catalogs

## Phase 2: Runtime Translator Loading

1. Update `src/main.cpp` to install `QTranslator` during startup.
2. Locale resolution order:
   1. user-selected language in settings
   2. system locale fallback
   3. default to `en`
3. Load `silent_<locale>.qm` before constructing the main window.
4. Add fallback behavior:
   - if translation file is missing/unloadable, continue in source language
   - log diagnostic information for troubleshooting
5. Persist selected language in app settings.

## Phase 3: Migrate UI Strings to Translatable APIs

1. Replace hardcoded user-facing strings with `tr(...)` where classes inherit `QObject`.
2. Use `QCoreApplication::translate(...)` in non-`QObject` contexts.
3. Convert dynamic strings to placeholders:
   - `tr("Inputs (%1 coins selected)").arg(count)`
   - `tr("Fee: %1").arg(value)`
4. Add translator comments where domain meaning may be unclear:
   - sats, sats/vB, coin selection, address reuse warning, etc.
5. Keep non-display protocol tokens unlocalized unless they are strictly UI labels.

## Phase 4: Retranslation Support (Live Language Switching)

1. Implement `changeEvent(QEvent::LanguageChange)` for top-level and major screen widgets.
2. Move text assignment into `retranslateUi()`-style methods.
3. Ensure retranslation updates all relevant text:
   - window titles
   - tab labels
   - placeholders
   - buttons
   - modal text
4. Even if live switching is deferred, preserve this structure for maintainability.

## Phase 5: Rust/FFI Error Boundary Hardening

1. Identify where Rust English error strings are surfaced directly in UI.
2. Introduce a C++ mapping layer from Rust machine-readable error kinds/codes to localized UI messages.
3. For each error path:
   - show localized user-friendly summary
   - optionally include raw technical details in secondary/debug context
4. Keep a safe fallback for unknown error types.

## Phase 6: Locale-Aware Formatting

1. Use `QLocale` for display formatting (numbers, grouping, date/time where used).
2. Keep internal transaction, amount parsing, and protocol serialization locale-invariant.
3. Define and document Bitcoin display rules:
   - localized labels/units
   - locale-aware numeric rendering
   - txids/addresses/outpoints remain raw and unchanged

## Phase 7: QA and CI Guardrails

1. Add CI/dev checks for translation lifecycle:
   - ensure translation targets build
   - optionally detect newly introduced hardcoded UI literals (with allowlist)
2. Manual test matrix:
   - default locale startup
   - non-English startup
   - language persistence after restart
   - send flow (including confirm/error modals)
   - account switching
   - settings save/load
3. Validate layout resilience with longer strings:
   - button sizing
   - label wrapping
   - modal dimensions

## Phase 8: Documentation and Contributor Workflow

1. Add concise i18n contributor guidance:
   - how to mark strings
   - how to update `.ts`
   - how to produce `.qm`
2. Maintain a domain glossary for translation consistency:
   - Address, Inputs, Outputs, Unconfirmed, Broadcast, Fee, sats, sats/vB, etc.
3. Add release checklist item to verify updated translations are packaged.

## Suggested Implementation Order

1. Translation pipeline in `CMakeLists.txt`.
2. Translator loading and language persistence in `src/main.cpp` + settings path.
3. Top-level UI (main window, tabs, global navigation).
4. `Send` screen + related modals (highest string volume/risk).
5. Remaining screens (`Receive`, `History`, `Settings`, `StatusBar`).
6. Rust error-code mapping improvements.
7. Live language switch support and final QA sweep.

## Definition of Done

- All user-visible Qt strings are translatable.
- At least one non-English locale is loadable end-to-end.
- Primary error flows show localized UI messages (not raw backend English only).
- Build/release reliably generates and bundles `.qm` artifacts.
- Selected language persists across restarts.
- Core workflows pass smoke tests in source and translated locales.
