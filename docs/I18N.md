# Silent i18n Workflow

## Translation Pipeline

- Source strings are defined in translator-friendly `.lang` catalogs.
- Qt `.ts` files are generated during the build into the build directory.
- Build-time translation compilation produces `.qm` files when `Qt6LinguistTools` is
available.
- Runtime loads translations from embedded Qt resources.

## Catalog Format

Primary editable files:
- `i18n/silent_en.lang`
- `i18n/silent_es.lang`

Line format:
- English source catalog (`silent_en.lang`): `id => "English"`
- Other language catalogs: `id => "English" => "Translation"`

Notes:
- Use `\n` for line breaks.
- Lines starting with `#` are comments.
- Only one or two quoted values are allowed per line.
- For two-value lines, the second value is the runtime translation.
- `NONE` is a reserved translation placeholder. If the selected runtime value is `NONE`, the entry is omitted from generated `.ts` and falls back to English.
- Multiline form is allowed and must end with `;;` on the last value line:
  - `id`
  - `=> "English"`
  - `=> "Translation";;`
- Lines longer than 85 chars are auto-written in multiline form by sync/add scripts.

## Adding a New Language

- Preferred path: `just add-lang <locale> <autonym>`
  - Example: `just add-lang de Deutsch`
- Add a new catalog file: `i18n/silent_<locale>.lang`.
- The command creates `i18n/silent_<locale>.lang` from English and adds the selector
item in `src/screens/Settings.cpp`.
- Build generation in `CMakeLists.txt` auto-discovers `i18n/silent_*.lang`, so no
per-language CMake edits are required.
- Add language label IDs in catalogs when needed:
  - `language-english`, `language-french`, `language-german`, `language-italian`,
`language-portuguese`, `language-spanish`
- Rebuild so generated `.ts` and embedded `.qm` include the new locale.
- Sync locale files from English source: `python3 scripts/i18n/sync_lang.py --lang-dir i18n`
- CI check mode (fails if sync is needed): `python3 scripts/i18n/sync_lang.py --lang-dir i18n --check`

## Adding a New ID

- Use `just add-i18n <id> <english-text>`.
- This appends `id => "English"` to `silent_en.lang`.
- It also appends `id => "English" => "NONE"` to every non-English `.lang` file.
- `NONE` lets you grep untranslated entries before release while keeping runtime fallback to English.

## Language Selection

- Startup locale resolution order:
  1. forced `en` at startup
  2. runtime selection via Settings (not persisted yet)
  3. persistence will move to Rust config later
- Settings screen includes a Language selector (`English`, `Francais`, `Italiano`,
`Deutsch`, `Portugues`, `Espanol`) and `Apply` action.

## ID-Based Translation Macros

- `TR("id")` - singular ID-based translation.
- `TRN("id", n)` - plural ID-based translation.

These macros provide stable translation keys that are independent of the English
copy.

## Relationship With Qt `tr(...)`

Qt `tr(...)` is strongly discouraged for product UI copy.

Use:
- `TR/TRN` for governed, stable-ID product copy.
- `tr(...)` only when there is a strong reason and `TR/TRN` cannot be applied cleanly.

## ID Rules

When using ID-based translations, IDs must use:
- lowercase unicode letters
- digits
- hyphen (`-`)

Examples:
- `delete-wallet`
- `settings-network-mismatch`

IDs used through `TR/TRN` must exist in the generated English TS catalog.

## CI Enforcement (Pragma Log Driven)

CI extracts used IDs from compiler pragma output:
- `I18N_ID:<id>` for `TR`
- `I18N_IDN:<id>` for `TRN`

### Fail conditions

- Used `TR` ID missing from generated `silent_en.ts`.
- Used `TRN` ID missing from generated `silent_en.ts`.
- `TRN` ID mapped to non-plural catalog entry.
- `TR` ID mapped to plural-only catalog entry.

### Warning conditions

- IDs present in generated `silent_en.ts` but unused in the build log.
